\section{Introduction}
Le Developer Manual décrit l'architecture mécanique, électronique et logicielle complexe du robot.
L'Agri-Scout est propulsé par ROS 2 Jazzy, OpenCV et um pont de communication dynamique avec Arduino.

\section{Architecture Mécanique}
Comme défini dans le Hardware Maintenance Manual, des supports personnalisés ont été conçus à partir de la structure de base du robot pour accueillir les capteurs et l'électronique. Le robot étant destiné à opérer en milieu agricole, il est équipé d'un capteur pour extraire les paramètres du sol. Pour que le capteur puisse fonctionner, il doit pénétrer le terrain ; à cet effet, une sonde à vis a été conçue et couplée au châssis du robot. La sonde est actionnée par un moteur pas à pas NEMA. Compte tenu de la complexité de la conception et de la fabrication de la sonde, la partie mécanique du robot a été divisée en deux parties principales : la conception/fabrication de la sonde et les autres supports du robot.

\section{Architecture Électronique}
L'unité centrale de traitement du robot est um Raspberry Pi 4 couplé à un Arduino UNO. L'Arduino est responsable du contrôle des moteurs de toutes les roues, ainsi que du moteur pas à pas de la sonde à vis via le pont en H. L'Arduino garantit que l'envoi des commandes aux moteurs est rapide et efficace, évitant les retards qui pourraient survenir si le Raspberry Pi s'en chargeait. Le Raspberry Pi traite les données des capteurs, tels que la caméra et le LIDAR, et envoie les commandes de mouvement à l'Arduino.

Le Raspberry Pi et l'Arduino ont été fixés sur le châssis supérieur du robot. Le pont en H connecté au moteur pas à pas a été fixé directement à l'Arduino pour faciliter la connexion et éviter l'usage de câbles longs. La figure \ref{fig:eletronica} montre la disposition des composants.

\subsection{LIDAR RPLiDAR A1}
Le LIDAR utilisé est un RPLiDAR A1 de Slamtec. Ce capteur effectue un balayage à 360° à l'aide d'un faisceau laser infrarouge, mesurant les distances aux obstacles environnants avec une précision millimétrique. Il est connecté au Raspberry Pi via USB et communique à un débit de 115200 bauds. Dans le système, il est principalement utilisé comme dispositif de sécurité : le nœud ROS 2 surveille en permanence le secteur frontal (±30°) et déclenche un arrêt d'urgence si un obstacle se trouve à moins de 20 cm. Le LIDAR est monté en hauteur sur le support d'élévation imprimé en 3D afin de maximiser son champ de vue.

\subsection{Centrale Inertielle (IMU)}
La centrale inertielle (IMU) est un module SparkFun OpenLog Artemis, qui intègre un accéléromètre 3 axes et un gyroscope 3 axes. Elle est fixée sur le même support rotatif que le LIDAR, permettant d'aligner les référentiels des deux capteurs. Bien que l'IMU ne soit pas encore pleinement exploitée dans la version actuelle du logiciel, elle est intégrée à l'architecture matérielle afin de permettre de futures améliorations, notamment pour la détection d'inclinaison du terrain ou la correction de trajectoire.

\subsection{Webcam USB}
La webcam est une caméra USB standard, utilisée comme capteur de vision principale du robot. Elle est gérée par le pilote ROS 2 \texttt{v4l2\_camera}, qui publie les images sur le topic \texttt{/image\_raw}. La résolution de capture est configurée à 320×240 pixels en format MJPEG, afin de réduire la charge de traitement sur le Raspberry Pi. Le nœud de contrôle redimensionne ensuite chaque trame à 160×120 pixels pour l'analyse par vision par ordinateur (détection de ligne et de marqueurs de récolte).

\section{Architecture Logicielle}

Le logiciel du robot est organisé autour d'un nœud ROS 2 central écrit en Python, d'un firmware Arduino et d'un ensemble de scripts utilitaires. Les fichiers sources sont regroupés dans deux répertoires principaux du dépôt :
\begin{itemize}
    \item \texttt{python\_control/} — scripts Python (nœuds ROS 2 et utilitaires autonomes)
    \item \texttt{arduino\_firmware/} — code source Arduino
\end{itemize}

\subsection{Nœud Principal : \texttt{agri\_line\_follower.py}}
Ce nœud agit comme le « cerveau » du robot en mission autonome. Il intègre trois flux asynchrones : la caméra V4L2 (topic \texttt{/image\_raw}), le scanner LIDAR (topic \texttt{/scan}) et la liaison série UART vers l'Arduino. La machine à états interne comporte deux modes principaux :
\begin{itemize}
    \item FOLLOWING : suivi de ligne blanche par vision, avec un correcteur PD pour le guidage latéral.
    \item HARVESTING : séquence automatique de descente, collecte de données et remontée de la sonde.
\end{itemize}
Afin d'éviter une surcharge du Raspberry Pi, plusieurs contraintes de performance sont appliquées :
\begin{itemize}
    \item OpenCV Mono-Core : \texttt{cv2.setNumThreads(1)} limite l'utilisation à un seul cœur CPU.
    \item Downscaling : l'image est redimensionnée à 160×120 pixels avant traitement.
    \item Throttling : les trames de débogage publiées sur \texttt{/agri\_debug} sont émises à raison d'une trame sur 15.
\end{itemize}
Un watchdog interne surveille la réception des images et ordonne l'arrêt d'urgence si aucune donnée n'est reçue pendant plus de 15 secondes. La récolte périodique automatique est également déclenchée toutes les 30 secondes via un timer ROS 2.

\subsection{Nœud de Démonstration : \texttt{agri\_demo\_hunter.py}}
Ce nœud alternatif a été développé pour les démonstrations. Au lieu de suivre une ligne, il recherche un objet coloré (par défaut rouge/orange) dans le champ de vision de la caméra et s'en approche. Il dispose d'une machine à états similaire (SEARCHING, TRACKING, HARVESTING) et utilise également le LIDAR pour la détection d'obstacles. Il peut être lancé manuellement à la place de \texttt{agri\_line\_follower.py}.

\subsection{Téléopération : \texttt{ros2\_teleop.py}}
Ce nœud permet de contrôler le robot manuellement depuis un terminal via le clavier (touches W/A/S/D pour les déplacements, I/K pour la sonde, Espace pour l'arrêt). Les commandes sont publiées sur le topic ROS 2 \texttt{agri\_commands}. Il est utile pour les phases de test et de réglage.

\subsection{Lecture du Capteur de Sol : \texttt{read\_sensor.py}}
Script autonome (sans ROS 2) qui communique avec le capteur de sol NPK via le protocole Modbus RTU sur le port série \texttt{/dev/ttyUSB1}. Il lit en continu 8 paramètres (température, humidité, conductivité, salinité, azote, phosphore, potassium, pH) et les affiche dans le terminal. À l'arrêt (Ctrl+C), les données de la session sont sauvegardées dans un fichier CSV.

\subsection{Calibration de la Sonde : \texttt{calibrate\_probe.py}}
Utilitaire interactif permettant de calibrer mécaniquement le rail de la sonde. Il envoie des commandes de pas au stepper via la liaison série et affiche la position cumulée (en pas), ce qui permet de déterminer le course maximale sans risque de détériorer la mécanique.

\subsection{Fichiers de Lancement ROS 2}
Deux fichiers de lancement (launch files) orchestrent le démarrage du système :
\begin{itemize}
    \item \texttt{demo.launch.py} — lance simultanément le nœud de caméra V4L2, le nœud LIDAR RPLiDAR et le nœud de suivi de ligne (\texttt{agri\_line\_follower.py}).
    \item \texttt{calibration.launch.py} — lance uniquement la caméra pour les procédures de calibration de l'optique.
\end{itemize}

\subsection{Firmware Arduino (\texttt{main\_firmware.ino})}
Dans \texttt{main\_firmware.ino}, l'Arduino agit comme un esclave de bas niveau chargé du contrôle temps réel des actionneurs. Il reçoit des paquets binaires sécurisés (protocole avec octet de début \texttt{0x3C}, commande, valeur, checksum XOR, et octet de fin \texttt{0x3E}) et exécute les actions correspondantes :
\begin{itemize}
    \item \texttt{F} / \texttt{B} : avance / recul
    \item \texttt{L} / \texttt{R} : virage gauche / droite
    \item \texttt{D} / \texttt{U} : descente / remontée de la sonde (stepper)
    \item \texttt{X} : arrêt complet
\end{itemize}
Un mécanisme de watchdog matériel coupe les moteurs si aucune commande n'est reçue pendant plus d'une seconde, garantissant la sécurité en cas de perte de communication. Pour migrer vers une commande différentielle directe et éviter le freinage ABS des ESC RC, le Python envoie désormais des valeurs de vitesse absolues, sans jamais inverser brusquement le sens.

\section{Compilation et Modification}
Pour recompiler le nœud ROS 2 :
\begin{verbatim}
cd ~/AGRI-Scout_Modif
colcon build --packages-select <votre_package>
\end{verbatim}

Pour modifier le code Arduino, utilisez l'IDE Arduino sur le code source dans le dossier \texttt{arduino\_firmware}. Le Raspberry Pi dispose d'arduino-cli installé, ce qui permet la compilation et le téléversement du firmware directement depuis le terminal du Raspberry Pi. Pour faciliter l'opération, le fichier \texttt{record.sh} a été créé ; il permet la compilation et l'envoi en lançant simplement la commande suivante :
\begin{verbatim}
bash record.sh <nom_dossier>
\end{verbatim}

Par exemple, si l'utilisateur modifie le code \texttt{main\_firmware.ino}, la commande sera :
\begin{verbatim}
bash record.sh main_firmware
\end{verbatim}
